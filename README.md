# magic_map

Declaration-site, fallible struct/enum mapping for Rust вАФ automap between
types you **don't own** (DB rows, prost-generated protos, OpenAPI DTOs) with
strict, compile-checked conversions.

```rust
use magic_map::{magic_map, MapInto};

// The mapping is a standalone declaration вАФ not an attribute on the type.
magic_map!(db::License => dtos::LicenseResponse {
    devices_used: 0,                          // absent from the source
    license_type: src.license_type == "pro",  // custom expression
});
// Every other field auto-fills from the same-named source field through a
// fallible leaf funnel: StringвЖФUuid, DecimalвЖФf64, Option/Vec wrappers,
// nested mapped enums/structsвА¶ and a typo'd or newly-added field is a
// COMPILE error, not a silently-unmapped field.

let dto: dtos::LicenseResponse = license.map_into()?;
```

## Why another mapper?

Every popular Rust mapping crate ([`o2o`], [`struct_convert`], [`derive_more`])
is **annotation-site**: you put attributes on a struct you own. That collapses
in the most common real-world layering вАФ proto types generated by
[prost-build], rows generated by an ORM, DTOs generated from an OpenAPI spec.
There is no struct to annotate, and the orphan rule blocks `From` impls
between two foreign types anyway.

`magic_map` flips the model:

- **The mapping lives at the call site** (your `mappers.rs`), so the two
  sides stay completely decoupled вАФ the proto crate never learns about the
  DB crate and vice versa.
- Types opt in with a single **content-free derive** (`#[derive(MagicMap)]`)
  that records only the type's own field names. For generated code, inject it
  through codegen config вАФ e.g. prost-build's `type_attribute(".", ...)` over
  the whole proto tree.
- The **fn form** generates a plain function instead of a trait impl, so even
  foreignвЖТforeign mappings (dbвЖТproto in a neutral service crate) work.
- Everything is **fallible and strict by default**. Lossy conversions return
  `Err(MappingError)` instead of inventing a default; unmatched enum variants
  and absent fields are compile errors.

|                          | magic_map | o2o | derive_more | frunk |
|--------------------------|-----------|-----|-------------|-------|
| mapping declared away from the types | вЬЕ | вЭМ attributes on the type | вЭМ | вЭМ |
| both sides generated/foreign | вЬЕ (derive via codegen config, or fn form) | вЭМ must annotate one side | вЭМ | вЪ†пЄП needs `LabelledGeneric` derive |
| fallible-first, strict leaves | вЬЕ | вЪ†пЄП opt-in `try_from` | вЭМ | вЭМ infallible |
| compile error on unmapped field | вЬЕ | вЬЕ | вЬЕ | вЬЕ |
| optionality adaptation (`Option<T>`вЖФ`T` with model defaults) | вЬЕ | вЭМ | вЭМ | вЭМ |

## Quick start

```toml
[dependencies]
magic_map = { version = "0.1", features = ["uuid", "chrono", "decimal"] }
```

```rust
mod db {
    #[derive(magic_map::MagicMap)]
    pub struct User {
        pub id: uuid::Uuid,
        pub name: String,
        pub age: i32,
        pub joined: chrono::DateTime<chrono::Utc>,
    }
}

mod dtos {
    #[derive(magic_map::MagicMap)]
    pub struct UserResponse {
        pub id: String,     // Uuid вЖТ String automaps (uuid leaf)
        pub name: String,
        pub age: i64,       // i32 вЖТ i64 widens losslessly, automaps
        pub joined: String, // DateTime<Utc> вЖТ rfc3339 String (chrono leaf)
    }
}

mod mappers {
    use magic_map::magic_map;
    // All four fields automap вАФ the declaration is empty.
    magic_map!(super::db::User => super::dtos::UserResponse);
}

use magic_map::MapInto;
let dto: dtos::UserResponse = user.map_into()?;
```

The derive is **content-free metadata**: it publishes the type's own field
names as a hidden macro next to the type and nothing else. It never references
another layer, so it is safe to apply blanket-style to every model, DTO, and
generated proto in a workspace.

## Grammar tour

### impl form

Generates `impl MapFrom<Src> for Dest`. Legal when your crate owns `Src` or
`Dest` (the usual dbвЖТdto / dtoвЖТdb case):

```rust
magic_map!(db::License => dtos::LicenseResponse {
    devices_used: 0,                          // dest field absent from src
    license_type: src.license_type == "pro",  // expression over `src`
});
```

### fn form

Generates a plain `pub fn name(src: Src) -> Result<Dest, MappingError>`. Use
it when **neither** type is local (dbвЖТproto in a service crate) and the orphan
rule forbids any impl:

```rust
magic_map!(pub fn equipment_to_proto: db::Equipment => proto::Equipment);
```

### `let` preludes

Shared derivations that feed more than one destination field:

```rust
magic_map!(pub fn span_stats: Span => SpanStats {
    let width = src.end - src.start;
    width: width,
    midpoint: src.start + width / 2,
});
```

### Tuple sources

`impl MapFrom<(A, B, ...)> for Dest` вАФ call sites do
`(a, b).map_into()?`. Plain non-generic struct elements are **open**: they
must derive `MagicMap` and their fields join the auto-match. Generic types,
references and primitives are **opaque**: reachable only as `src.N` in
overrides. A destination field found in exactly one open element automaps;
found in several (or none) is a compile error asking for an override:

```rust
magic_map!((db::License, db::Owner, i64) => dtos::LicenseCard {
    id: src.1.id,            // `id` exists in both License and Owner вЖТ explicit
    devices_used: src.2 as i32,
});
// `name` automaps from Owner, `max_devices` from License вАФ each is
// unambiguous.
```

A single by-reference source uses a 1-tuple: `magic_map!((&Ctx,) => Dest {...})`.

### Enum mappings

Variant-by-name, generated **per source variant** вАФ so extra destination
variants are fine (never produced), but an unmatched source variant is a
compile error. `Src => Dest` rename pairs handle vocabulary drift, and the
proto3 zero variant `Unspecified` folds to the destination's declared
`#[default]`:

```rust
magic_map!(db::Status => dtos::StatusDto);          // same names
magic_map!(pub fn kind_to_wire: DbKind => WireKind {
    Integer => Int32,                                // explicit renamesвА¶
    Text    => String,                               // вА¶the rest pair by name
});
```

### The `..Default::default()` optionality adaptor

For sparse create/update models, a trailing `..Default::default()` makes the
mapping default-tolerant (the destination must implement `Default` вАФ put
business defaults **on the model**, e.g. with [`smart-default`]):

```rust
magic_map!(dtos::CreateLicenseRequest => db::CreateLicense { ..Default::default() });
```

Three behaviors compose, picked automatically per field:

1. `Option<S>` source вЖТ required dest: unwrap through the funnel; `None`
   falls back to the **default instance's field value** вАФ every
   `unwrap_or(business_default)` line disappears.
2. `Option вЖТ Option`: stays a plain funnel, so `None вЖТ None` вАФ an optional
   destination never gets a value invented.
3. Plain source вЖТ `Option<U>` dest: funnel and wrap in `Some` ("set"
   semantics). If `None` means "don't touch" on your patch models, keep
   explicit wraps there.

Destination fields absent from every source come from `Default::default()`.
Use the trailer deliberately: it trades the missing-field compile error for
defaulting.

[`smart-default`]: https://crates.io/crates/smart-default

## Leaves

A *leaf* is a `MapFrom` impl for a known type pair. Identities for primitives
and `String` ship always; third-party leaves are feature-gated:

| feature   | leaves |
|-----------|--------|
| `uuid`    | `Uuid` identity, `StringвЖФUuid` (strict parse) |
| `chrono`  | date/time identities, `DateTime<Utc>вЖФString` (rfc3339), `NaiveDateвЖФString` (ISO-8601) |
| `decimal` | `Decimal` identity, `DecimalвЖФf64` / `DecimalвЖФString` (strict вАФ NaN/вИЮ error) |
| `json`    | `serde_json::Value` identity |
| `full`    | all of the above |

Lossless integer widenings (`u8вЖТu16вА¶i64`, `i32вЖТi64`, `f32вЖТf64`) automap;
narrowing stays an explicit `as` cast in an override, where the lossiness is
visible.

### Your own leaves

Declare them in the crate that owns the type вАФ they then automap everywhere:

```rust
magic_map::map_identity!(MyEnum);   // MyEnum вЖТ MyEnum (modelвЖТmodel moves)
magic_map::map_display!(MyEnum);    // MyEnum вЖТ String (pairs with strum Display)
magic_map::map_parse!(MyEnum);      // String вЖТ MyEnum, strict (strum EnumString)

// Or hand-write any pair:
impl magic_map::MapFrom<MyWireTimestamp> for chrono::DateTime<chrono::Utc> {
    fn map_from(src: MyWireTimestamp) -> Result<Self, magic_map::MappingError> {
        /* strict conversion */
    }
}
```

## prost / generated-code recipe

In `build.rs`, plant the derive on every generated type:

```rust
prost_build::Config::new()
    .type_attribute(".", "#[derive(magic_map::MagicMap)]")
    .compile_protos(&["proto/models.proto"], &["proto/"])?;
```

Unsupported shapes (tuple structs, enums with payload variants) are a silent
no-op, so the blanket attribute is safe. If two same-named types exist in one
crate (e.g. `pkg_a.Sale` and `pkg_b.Sale`), disambiguate one's hidden export:
`#[magic_map(export = "PkgASale")]`.

Then map protoвЖФdb in a service crate with the fn form вАФ no crate ever
depends on the other's "shape".

## Error handling philosophy

Every mapping returns `Result<_, MappingError>`. Convert it **once** into
each layer's error type (`impl From<MappingError> for ApiError`) and bubble
with `?`. Mappers fail loudly; quarantining/skipping bad records is a
decision for the handler that owns the batch, not for the conversion.

## Limitations

- The generated code references the crate by its real name вАФ `magic_map`
  must be a direct dependency, not renamed.
- Two same-named `MagicMap` types in one crate collide on the hidden
  crate-root export вАФ rename one or use `#[magic_map(export = "...")]`.
- Destination types must have named fields (or unit variants); tuple structs
  are not supported as destinations.
